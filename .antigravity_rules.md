# Antigravity Project Rules & Behavioral Guidelines

## Behavioral Guidelines (LLM Mistakes Reduction)

### 1. Think Before Coding
- Don't assume. State assumptions explicitly.
- Surface tradeoffs. If multiple interpretations exist, ask before picking.
- If a simpler approach exists, propose it. Push back on overcomplication.
- If something is unclear, stop and ask.

### 2. Simplicity First
- Minimum code that solves the problem. Nothing speculative.
- No features, abstractions, or "flexibility" beyond what was requested.
- No error handling for impossible scenarios.
- Ask: "Would a senior engineer say this is overcomplicated?" If yes, rewrite.

### 3. Surgical Changes
- Touch only what you must.
- Don't "improve" adjacent code, comments, or formatting. Match existing style.
- If changes create orphans (imports/variables), remove them.
- Don't remove pre-existing dead code unless asked.
- Every changed line should trace directly to the user's request.

### 4. Goal-Driven Execution
- Define success criteria. Loop until verified.
- Transform tasks into verifiable goals.
- For multi-step tasks, state a brief plan: `[Step] -> verify: [check]`.

---

## Project-Specific Rules
- **GitHub Policy**: ONLY push to GitHub if the user explicitly orders it mentioning the skill `github-commit`. Without that specific mention, keep changes local.
- **Tech Stack**: Next.js, TypeScript, InsForge, Tailwind CSS (Vanilla CSS preferred for custom components).
- **Aesthetics**: Premium, modern design (glassmorphism, vibrant colors, dark mode, smooth animations). No placeholders.
- **Data Flow**: Always use `src/app/actions/` for server-side logic and database interactions via InsForge SDK.
